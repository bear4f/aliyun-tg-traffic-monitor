# Changelog

## 3.2.0

### 主面板改为只读总览
- 移除主面板上的直接开关机按钮——高风险操作全部移入机器详情页（保留二次确认）。机器按钮显示实时状态与用量（如「🟢 阿里云香港 · 36.7%」），点击进入详情。
- 用量百分比按风险着色（🟢 正常 / 🟡 接近阈值 / 🔴 达到阈值）。
- 数据新鲜度着色：距上次成功查询 ≤2 个周期 🟢、≤6 个周期 🟡、更久 🔴，一眼区分「流量安全」与「监控本身异常」。

### 用量趋势与触线预测
- 每小时采样用量历史（保留约 8 天），计算 24 小时与 7 日滚动日均，预测速度取两者较大值——突发跑量能更快反映到预测里。
- 面板卡片显示「🔮 预计月底 XX GB」与预计触线日期（如「⚠️ 约 07-28 触线」）；详情页分别显示 24 小时 / 7 日速率。
- 历史不足时回退到本月累计日均。

### 双阈值熔断
- 普通熔断线（`shutdown_percent`，默认 95%）需**连续两次**成功查询确认才关机，防止单次异常 API 数据或口径配置错误误关机。
- 新增紧急熔断线（`emergency_shutdown_percent`，默认 98%）：一次确认即立即关机，兜底超额费用风险。
- 详情页明确显示两条线、实际关机线对应的 GB 数、流量口径与数据来源 API；同一账号同口径被配置到多台机器时提示流量为账号级共享值。

### 事件记录与 Bot 健康
- 新增「📋 事件记录」页：保留最近 50 条关键事件——熔断触发/失败、手动开关机、阈值与额度修改、查询异常与恢复、账期切换、新账期自动开机。例行成功查询不记录。
- 全局设置页新增 Bot 运行时长与 24 小时 API 成功率统计。

## 3.1.4

- **读取超时 15 秒 → 30 秒**：实测存在真实的读取阶段慢响应（`read timeout=15`），监控场景宁可等慢答复也不要误报失败。最坏情况一轮完整重试约 2 分钟，仍远小于 300 秒检查周期。
- **重试加入随机抖动**：避免多台机器的重试在同一时刻扎堆请求同一端点。
- **开机 / 关机 / 重启不再自动重试**：修改类接口只发一次——若指令已生效但响应超时，盲目重试会把成功误报为失败（IncorrectInstanceStatus），重启甚至可能双重触发。查询类接口保持 3 次重试不变（符合阿里云官方建议）。

## 3.1.3

### 查询失败通知去噪：偶发波动全静默，过程只在面板体现
- **单次/偶发失败不再发通知**：新增 `error_notify_after_failures`（默认 3，可在配置中调整），连续失败达到该次数（默认约 15 分钟）才发一条「持续查询失败」通知。偶发的网络抖动只体现在面板卡片的「⚠️ 查询失败中（连续 N 次）」上，恢复后自动消失。
- **取消「已恢复」通知**：恢复时不再发消息，改为**自动撤回**之前发出的失败通知，聊天记录保持干净；面板回到正常状态即代表恢复。
- 全局设置页显示失败通知策略；持续失败期间的重复提醒仍受 1 小时冷却限制。

## 3.1.2

### 面板实时自动刷新
- 后台监控循环每轮检查结束后，会自动原地重绘最近一次打开的主面板消息：头部的「数据更新于」持续前进，一眼能确认监控还活着，不再需要手动点刷新。
- 只在面板停留在主视图时才自动重绘；进入机器详情、全局设置或等待输入时后台不会打扰。面板消息被删除或超过 Telegram 约 48 小时可编辑窗口后自动解绑，重新 /menu 即恢复。
- 头部新增检查频率显示（如「🛰️ 每 5 分钟自动检查」），有机器查询异常时追加「⚠️ N 台查询异常」。

### 查询失败的完整生命周期
- **失败不再覆盖数据**：查询失败时保留上次成功的用量读数，卡片在正常数据下方叠加「⚠️ 查询失败中（连续 N 次，最近 HH:MM）」，不再变成一张只有报错的卡片。
- **失败通知带上下文**：连续失败时通知包含连续次数与起始时间，并说明提醒频率上限。
- **恢复通知**：失败流结束（查询重新成功）时发送「✅ 查询已恢复」，附失败次数与持续时长——报错不再没有下文。
- **冷却按失败流重置**：恢复后如再次出现新的失败，立即通知，不再被上一轮的 1 小时冷却吞掉。

## 3.1.1

- **修复升级不生效**：install.sh 结尾的 `systemctl enable --now` 对已在运行的服务是空操作，升级后服务仍跑旧代码。现在检测到服务运行中会自动 `restart` 加载新版本。
- **修复偶发的「查询失败 … Read timed out. (read timeout=5)」**：这个报错来自 urllib3 的一个误导性文案——TLS 握手阶段超时会被它报成 "Read timed out"，数值取的是连接超时（5 秒）。到 `cdt.aliyuncs.com` 的握手在网络抖动时偶尔超过 5 秒，且三次重试间隔太密（1.5s/3s），全部撞在同一波抖动里。现将连接超时放宽到 10 秒，重试间隔拉长为 3s/6s。
- **面板「更新于」改为绝对时间**：主面板头部与机器详情中的「更新于 x 分钟前」改为按配置时区显示 `2026-07-20 08:20` 这种格式，更直观。

## 3.1.0

### 修复按钮点击无反应
- **阿里云 SDK 超时单位纠错**：`set_connect_timeout(5000)` 的单位是秒不是毫秒（两个参考项目也抄错了），实际连接超时 83 分钟、读超时 4 小时——网络抖动时一次查询能把工作线程挂住几十分钟。改为 5 秒 / 15 秒。
- **并发处理更新**：python-telegram-bot 默认串行处理更新，一个慢操作会让之后所有按钮点击排队、Telegram 等不到应答就放弃显示。启用 `concurrent_updates`，慢查询不再冻结其它按钮。

### 操作动态反馈
- 点击「刷新全部 / 刷新」后，消息立即切换为「⏳ 正在查询 N 台机器……」过渡态，完成后回到面板并显示「✅ 已刷新 · 时:分:秒」。
- 开机 / 关机 / 重启执行时依次显示「⏳ 正在核对当前流量」「⏳ 正在发送指令」，结果行带时间戳。
- 所有面板操作都能明确看到"点了、在做、做完了"三个阶段。

## 3.0.4

- 修复一键安装的关键缺陷：在旧版源码目录（如之前的 git clone）里执行 `curl … | bash` 时，自举检测误信了当前目录下的旧文件，导致安装的仍是旧版本、上一版的修复全部没生效。现在管道执行一律强制下载最新 main，当前目录内容不再被信任；`bash install.sh` 直接运行则照常使用本地源码。
- 安装完成信息回显当前版本号，便于确认没有装到旧版。

## 3.0.3

- 修复首次配置向导的死循环：向导曾提示「向 Bot 发送 /id 获取 User ID」，但首次安装时 Bot 服务尚未启动，/id 不可能有回复，用户无从得知自己的数字 ID。现在向导用刚验证过的 Token 直接调用 getUpdates：向 Bot 随便发一条消息，按回车即自动读取并回填 User ID。
- Ctrl+D / 输入流关闭不再抛 Traceback，改为干净取消（退出码 130）。
- User ID 输入兼容全角逗号（，、；）与空格，多次输入自动去重；报错文案更明确。

## 3.0.2

- 新增 curl / wget 一键安装：`curl -fsSL …/install.sh | sudo bash`。install.sh 检测到不在源码目录运行时，自动下载最新 main 压缩包并从解压目录重新执行；管道场景下 stdin 已被脚本本身占用，首次配置向导自动改从 /dev/tty 读取，交互不受影响。升级复用同一条命令，配置保留并备份。

## 3.0.1

### EIP / 实例安全加固
- **移除 `StopCharging`（节省停机）**：该模式会回收实例的固定公网 IP。关机硬编码为 `KeepCharging` 且不可配置；旧配置中的 `StopCharging` 在加载时被强制改写，向导与 doctor 都会提示。
- RAM 策略增加**显式 Deny**：`ecs:DeleteInstance(s)`、`ecs:ModifyInstanceChargeType`、`vpc:ReleaseEipAddress`、`vpc:UnassociateEipAddress`、`vpc:DeleteEipAddress`。Deny 压过一切 Allow，即使 RAM 用户被误加宽权限也删不了机、放不掉 EIP。
- `aliyun-monitor doctor` 新增「实例 / EIP 安全」检查节：列出全部可调用 API 白名单、确认 KeepCharging、检测旧配置残留。
- Telegram 实例面板显示「🔒 停机保留公网 IP」；帮助页新增安全边界说明。

### 面板信息补强
- 新增消耗速度预测：按本月日均用量推算距离熔断线的天数，Telegram 主面板、实例面板与终端状态栏均显示（如「日均 8.3 GB · 约 4 天后触线」）；预测落在账期重置之后时显示「本账期无触线风险」。
- 「本账期已触发熔断」状态从 state.json 显性化到面板：主面板熔断行标注（本账期已触发🛑），实例面板说明新账期自动解除。

## 3.0.0

### Telegram 面板
- 改为单条消息原地刷新，层级导航（主面板 / 实例 / 全局设置 / 管理员 / 帮助），全部带 `🏠 返回`。
- 主面板展开显示每台机器的进度条、用量、剩余、状态、阈值与开关，两台机器无需下钻。
- 开关机按钮随实例状态自动切换（运行中显示关机，已关机显示开机）。
- 阈值支持 ±1% / ±5% 快调。
- 月度额度、分级提醒线、每日汇总时间、检查间隔、开机确认线、管理员列表均可在面板内修改。
- 新增文本输入流程：提示后直接发消息，程序校验后自动删除该消息并原地重绘面板。
- 唯一管理员不可被移除，避免把自己锁在门外。
- 每日汇总在机器数超过一页时按页发送，不再只发第一页。

### 终端面板
- `setup.py` 更名为 `panel.py`（避免与 Python 打包约定混淆），安装器会清理旧文件。
- 顶部彩色实时状态栏：服务状态、账期与剩余天数、每台机器的进度条与数据新鲜度。
- **录入 AccessKey 后立即调用阿里云 API 验活**，分别探测实例状态与流量读取，失败时给出具体排查方向。
- 新增 `aliyun-monitor doctor` 一键自检：配置结构、文件权限、systemd 状态、时区与账期、Telegram Token、每个账号的阿里云 API 连通性。
- 配置 Telegram 时调用 `getMe` 校验 Token 并回显 Bot 用户名。
- 时区输入即时校验。
- 菜单重组为两层，机器管理独立成子菜单。

### 修复
- `python-telegram-bot==22.8` 实际要求 Python 3.10+，在 Debian 11（Python 3.9）上安装必然失败。requirements 改用环境标记按版本分流，install.sh 增加 Python 版本预检。
- 刷新时若内容未变，Telegram 返回 `message is not modified` 会抛异常中断处理，现已吞掉该特定错误。
- 同一个回调被 `answer()` 两次时，第二次的提示（如「实例不存在」）不会显示给用户，现已改为单次应答。
- 重启后面板显示「尚未查询」直到下一轮循环，现从 `state.json` 恢复上次读数。
- 单实例刷新后未持久化快照，重启即丢失，现已统一落盘。
- 新增全局错误处理器，未捕获异常不再静默丢失。

### 其它
- 抽出 `common.py`，Telegram 与终端两侧共用同一套阿里云 API、配置模型与格式化逻辑。
- 向导会算出关机阈值对应的十进制 GB，并在超过 200 GB 免费额度时告警（GB / GiB 口径差异）。
- 默认关机阈值由 95% 调整为 90%，在两种换算口径下都留有余量。
